9:T1383,<h1>gstack</h1>
<blockquote>
<p><strong>한 줄</strong>: Y Combinator CEO Garry Tan이 자기 셋업을 그대로 공개한 슬래시 커맨드 23개. <strong>CEO·eng manager·designer·QA lead·security officer·release engineer</strong> 등 23개 페르소나로 변신하는 Claude Code 플러그인.</p>
</blockquote>
<ul>
<li><strong>Repo</strong>: <a href="https://github.com/garrytan/gstack">garrytan/gstack</a></li>
<li><strong>저자</strong>: Garry Tan (Y Combinator)</li>
<li><strong>라이선스</strong>: MIT</li>
<li><strong>설치</strong>: <code>git clone --single-branch --depth 1 https://github.com/garrytan/gstack.git ~/.claude/skills/gstack &#x26;&#x26; cd ~/.claude/skills/gstack &#x26;&#x26; ./setup</code></li>
</ul>
<h2>왜 이걸 쓰나</h2>
<p>혼자서 팀 출하 속도를 내려면 <strong>각 단계마다 다른 관점이 필요하다</strong>. 기능을 짜는 손과, 그걸 비판하는 눈, QA 도는 손, 보안 감사하는 눈은 한 사람의 머리에서 동시에 돌리기 어렵다.</p>
<p>gstack은 이걸 <strong>슬래시 커맨드 = 페르소나</strong> 1:1 매핑으로 푼다. <code>/plan-ceo-review</code>를 부르면 Claude가 "야심차게 다시 생각하는 CEO" 모드로 plan을 깬다. <code>/plan-eng-review</code>를 부르면 "아키텍처·엣지케이스 잡는 eng manager" 모드. <code>/qa</code>를 부르면 헤드리스 브라우저로 실제 사이트를 테스트하는 QA. <em>같은 모델이지만 매번 다른 스탠스</em>로 검토한다.</p>
<h2>우리 워크스페이스에서 자주 쓰는 것</h2>
<p><strong>계획 단계</strong>:</p>
<ul>
<li><code>/office-hours</code> — YC식 6개 강제 질문(수요 현실, 가장 좁은 wedge, 미래 적합성)으로 막연한 아이디어 압박</li>
<li><code>/plan-ceo-review</code> — 스코프 확장/축소 4모드. "더 야심차게" 또는 "본질만 남기고 다 자르자"</li>
<li><code>/plan-eng-review</code> — 아키텍처·데이터 흐름·엣지케이스·테스트 커버리지 확정</li>
<li><code>/plan-design-review</code> — 디자인 차원별 0-10 채점 후 plan 보강</li>
<li><code>/autoplan</code> — 위 4개 리뷰를 자동 결정 원칙으로 연속 실행</li>
</ul>
<p><strong>구현/검증 단계</strong>:</p>
<ul>
<li><code>/design-shotgun</code> — AI가 디자인 변형 여러 개 생성 → 비교 보드 → 피드백</li>
<li><code>/design-html</code> — 승인된 mockup을 production HTML/CSS로 변환 (텍스트 reflow, 동적 레이아웃)</li>
<li><code>/qa</code> — 헤드리스 브라우저로 실제 QA + 버그 자동 수정·재검증 루프</li>
<li><code>/review</code> — diff 검토 (SQL 안전성, LLM trust boundary, 조건부 부작용)</li>
<li><code>/codex</code> — OpenAI Codex CLI 래퍼. <strong>두 번째 의견</strong> 얻을 때 (코드리뷰·challenge·consult 3모드)</li>
<li><code>/cso</code> — Chief Security Officer 모드. OWASP + STRIDE + 공급망 + LLM 보안</li>
</ul>
<p><strong>출하 단계</strong>:</p>
<ul>
<li><code>/ship</code> → <code>/land-and-deploy</code> → <code>/canary</code> — PR → 머지·CI·배포 → 라이브 모니터링</li>
<li><code>/document-release</code> — 출하 후 문서 동기화 + CHANGELOG 다듬기</li>
</ul>
<p><strong>기타</strong>:</p>
<ul>
<li><code>/browse</code> — 헤드리스 브라우저(~100ms/명령). 사이트 도그푸딩·반응형 테스트</li>
<li><code>/investigate</code> — 4단계 근본 원인 추적 (조사→분석→가설→구현)</li>
<li><code>/retro</code> — 주간 엔지니어링 회고 (커밋 분석, 트렌드)</li>
<li><code>/careful</code> / <code>/freeze</code> / <code>/guard</code> — 파괴적 명령 가드, 디렉터리 잠금</li>
</ul>
<h2>시작하는 법 (가장 작은 예시)</h2>
<pre><code># 학생: 새 사이드 프로젝트 아이디어가 막연할 때
/office-hours

# 6개 질문에 답하고 나면 design doc 한 장이 떨어짐
# 그다음
/autoplan

# 자동으로 CEO/eng/design 리뷰가 연속 실행되고
# 결정 원칙으로 plan 확정
# 그다음 코드 짜고
/qa &#x3C;staging URL>

# 버그 잡혔으면 자동 수정·재검증 → 깨끗하면
/ship
</code></pre>
<h2>함정</h2>
<ul>
<li>23개를 다 외울 필요 없음. <em>오피스아워 → 오토플랜 → QA → 십</em>만 알아도 80% 커버.</li>
<li><code>/codex</code>는 OpenAI Codex CLI가 별도 설치되어 있어야 함. 우리 랩 셋업에는 이미 들어있음.</li>
<li>슬래시 커맨드는 <em>공유 셋업</em>이라 팀 모드(<code>./setup --team</code>)를 쓰면 repo 내 <code>.claude/</code>로 자동 동기화. 다만 우리 랩은 개인 셋업으로 충분.</li>
</ul>
<hr>
<blockquote>
<p>학부생이 <code>/office-hours</code> 한 번이라도 돌려보고 나서 자기 프로젝트 plan을 짜면, <strong>어떤 가정이 위험한지</strong> 스스로 보이기 시작함. 그 한 번이 첫 학습 포인트.</p>
</blockquote>
