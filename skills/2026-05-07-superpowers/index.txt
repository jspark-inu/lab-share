9:Te96,<h1>Superpowers</h1>
<blockquote>
<p><strong>한 줄</strong>: Claude Code(또는 Codex/Gemini/Cursor)가 코드부터 짜지 않게 강제하는 스킬 묶음. <em>"이거 만들어줘" → 즉시 코드</em>가 아니라 <strong>의도 → 스펙 → 계획 → TDD → 검증</strong> 순서를 자동으로 발동시킨다.</p>
</blockquote>
<ul>
<li><strong>Repo</strong>: <a href="https://github.com/obra/superpowers">obra/superpowers</a></li>
<li><strong>저자</strong>: Jesse Vincent (obra)</li>
<li><strong>설치</strong>: Claude Code 공식 마켓플레이스 — <code>/plugin install superpowers@claude-plugins-official</code></li>
</ul>
<h2>왜 이걸 쓰나</h2>
<p>기본 상태의 코딩 에이전트는 <strong>"빨리 답하고 싶어" 편향</strong>이 있다. 어설픈 한 줄 요청에도 바로 코드를 쓰기 시작하고, 뒤늦게 사용자가 "아 이거 말고…" 하면서 다시 짠다. 토큰만 태우고 결과는 어긋난다.</p>
<p>Superpowers는 이 편향을 <em>스킬 트리거</em>로 끊는다. 새 기능을 만들겠다고 말한 순간 <code>brainstorming</code> 스킬이 자동으로 깨어나서 <strong>무엇을 왜 만드는지부터</strong> 캐묻는다. 사용자가 의도를 정리하고 나면 그다음에 <code>writing-plans</code>가 따라붙고, 실행 단계에서는 <code>subagent-driven-development</code>가 서브에이전트를 분배한다.</p>
<p>핵심 슬로건은 <em>"If a skill applies, you do not have a choice. You must use it."</em> — 일종의 자기-규율 코드.</p>
<h2>우리 워크스페이스에서 자주 쓰는 것</h2>
<ul>
<li><strong><code>brainstorming</code></strong> — 새 기능·스킬·에이전트·제안서 만들기 시작 시 자동 진입. 의도·요구사항·디자인을 분리해서 묻는다.</li>
<li><strong><code>writing-plans</code></strong> — 멀티스텝 작업 전 체크리스트형 계획 작성. 후배 엔지니어도 따라할 수 있을 정도까지 명확하게.</li>
<li><strong><code>test-driven-development</code></strong> — 빨강(실패하는 테스트) → 초록(통과) → 리팩터 사이클 강제.</li>
<li><strong><code>systematic-debugging</code></strong> — 버그·테스트 실패·예외 만나면 임시방편 금지, 근본 원인까지 추적.</li>
<li><strong><code>verification-before-completion</code></strong> — "다 됐다"고 선언하기 전에 <em>작동 증명</em>을 요구.</li>
</ul>
<h2>시작하는 법 (가장 작은 예시)</h2>
<pre><code># 사용자
이 함수에 캐시 좀 붙여줘

# Superpowers 없는 에이전트
[즉시 LRU 캐시 코드 작성, 잘못된 invalidation 정책으로 ship]

# Superpowers 있는 에이전트
[brainstorming 발동]
"잠깐. 이 함수가 어떤 입력에 자주 호출되나요?
 stale data가 얼마나 위험한가요?
 캐시 invalidation 트리거가 뭐가 있나요?"
[답변 받은 후 → writing-plans → 그제서야 코드]
</code></pre>
<h2>함정</h2>
<ul>
<li>작은 수정에도 스킬이 발동하면 답답할 수 있다. <em>5초 안에 끝낼 수정</em>에는 명시적으로 "skill skip"이라고 알려줘도 OK (단, *"그냥 빨리 해"*가 습관이 되면 의미 없음).</li>
<li>TDD 스킬은 자동화가 어려운 도메인(시각 디자인, 디자인 토큰)에서는 적용하기 까다로움 — 이 경우 <code>frontend-design</code> 같은 다른 스킬에 위임.</li>
</ul>

<p>우리 랩의 <strong>거의 모든 코딩 세션이 이 스킬 위에서 돈다.</strong> Claude Code 처음 쓰는 학생이라면 <code>brainstorming</code>과 <code>verification-before-completion</code> 두 개만 알아도 토큰·시간 절약 효과 큼.</p>
