# 데이터과학과 인력사무소

> **한 줄**: 교수·행정 선생님의 *실제 문제*를 학과 학생이 *풀어보는* 자체 공모전 매칭 플랫폼.

## 동기 — 두 가지 막힘이 서로를 풀 수 있다

**학생 쪽 막힘**
- "데이터를 분석하고 싶은데 *진짜 문제*가 안 보인다."
- 캐글 같은 외부 데이터셋은 *문제 정의*가 이미 끝난 상태라 학습 효과가 제한적.
- 포트폴리오에 "내가 직접 발굴한 문제" 항목이 비어 있음.

**교수·행정 쪽 막힘**
- "개선하고 싶은 게 있는데 *직접 들여다볼 시간*이 없다."
- 학과 운영 중 발견한 의문(예: 학생 수강 패턴, 졸업 후 진로 추적, 행정 처리 병목)들이 *기록도 안 된 채* 사라짐.

이 두 막힘은 *서로의 답*이다. 다만 매칭하는 *자리*가 없다.

## 무엇을 만들 것인가 (예비 그림)

| 역할 | 행위 |
|---|---|
| **의뢰자** (교수·행정) | 내가 풀고 싶었으나 시간 없는 *진짜* 문제를 등록 |
| **풀이자** (학생) | 의뢰 목록에서 *관심 있는 1개*를 골라 풀이 제출 |
| **심사·발표** | 학기 1-2회 결과 발표회, 우수작 선정 |
| **인센티브** | (미정 — 결정 필요) 학점 / 장학금 / 포트폴리오 노출 / 채용 추천서 등 |

## 결정 필요 — 오픈 질문 3가지

### 1. 구현 형태 — 학과 사람들이 *많이 이용하게* 하려면

| 옵션 | 장점 | 약점 |
|---|---|---|
| **A. 별도 웹 플랫폼 신규** | 자유도 ↑, 학과 브랜드화 | 개발 시간 ↑, 운영 부담 |
| **B. lab-share에 새 카테고리 추가** | 즉시 가능, 디자인 일관성, 자동 배포 인프라 그대로 | lab-share가 *연구실용*이라 학과 전체에 노출 미묘 |
| **C. 노션 데이터베이스** | 5분 셋업, 모든 사람이 익숙 | 노션 계정 의존, 검색·통계 약함 |
| **D. 슬랙 채널 + 스프레드시트** | 가장 가벼움, 즉시 대화 가능 | 의뢰가 *흘러가버림*, 검색·아카이브 약함 |

**현재 직관**: B 또는 C가 빠른 시작. A는 사용자가 검증된 후.

### 2. 인센티브 구조 — *지속가능*하려면

핵심은 *양쪽 모두에게 의미 있는 보상*. 한쪽만 매력적이면 의뢰가 쌓이거나 풀이가 쌓이는 *비대칭 큐*가 됨.

**의뢰자(교수·행정) 측**
- 의뢰만 등록해도 *결과를 받을 가능성* 자체가 보상 (현재는 0)
- 학생 풀이를 *데이터과학과 결과물*로 학과 홍보·연차 보고서에 활용 가능
- (미정) 의뢰 빈도 ↑ 시 학과 차원 인정

**풀이자(학생) 측**
- 포트폴리오 1개 추가 (실제 학과 문제 + 발표 기록)
- (미정) 학점 연동 — 캡스톤·졸업과제 대체 인정?
- (미정) 우수작 → 장학금 / 채용 추천서 / 컨퍼런스 발표 기회
- (미정) 학생-학생 *팀 결성*도 가능하게?

### 3. 첫 문제 — 어떤 *시드 의뢰*로 시작할 것인가

학생들이 "이런 종류의 문제가 올라오는구나"를 첫 인상으로 갖게 하는 1-2개. 너무 어려우면 진입 장벽, 너무 쉬우면 가치 의심.

**후보** (아이디어 단계, 추가·교체 환영):
- 학과 학생 진로 패턴 분석 (졸업 후 5년 추적)
- 수강신청 데이터 → 트랙 제안 알고리즘
- 행정 문서 분류 자동화 (조교 시간 절감)

## 완료 기준 (예비)

- 첫 학기 안에:
  - 의뢰 **5건** 이상 등록
  - 학생 풀이 **3건** 이상 제출
  - 결과 발표회 **1회** 개최
- 두 번째 학기에:
  - 학생 *반복 참여율* > 30% (한 번 푼 학생이 다음 학기에 다시 참여)
  - 의뢰자 *반복 등록률* > 50%

## 다음 액션 (직속 task로 db.json에 등록)

1. **구현 형태 결정** — 4 옵션 비교 후 이번 주 안에 1개 선택
2. **인센티브 구조 설계** — 학과와 협의 가능한 범위 확인
3. **시드 의뢰 1-2건 직접 등록** — 의뢰 템플릿 검증
4. **MVP 운영 계획** — 첫 학기 일정 확정
